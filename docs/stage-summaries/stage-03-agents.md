## 1. What this stage did

Replaced Stage 2's mock `EchoEngine` with a real LangGraph agent loop calling the
Anthropic API. `POST /v1/chat/completions` now reasons, calls tools, observes the
results and answers, persisting conversation history to Postgres behind a Redis
read-through cache and reporting the model's own token counts.

The `CompletionEngine` seam held: the endpoint, the schemas and the SSE frame
format are unchanged.

### Migration tool chosen — and why

**Plain `.sql` files + a ~100-line runner in `shared/migrations.py`.** Not
Alembic. The codebase has no ORM (ADR 0005 rejected SQLAlchemy for the same
reason), and Alembic is SQLAlchemy's migration tool — adopting it means adding an
ORM to satisfy two `CREATE TABLE`s and then owning two incompatible ways to reach
Postgres. Its autogenerate feature, the main reason to pay that, diffs ORM models
we do not have. Forward-only: a `downgrade` is untested code that runs only
during an incident. Full reasoning and the rejected alternatives: **ADR 0007**.

### Redis caching strategy — and why

**Read-through cache; Postgres is the source of truth; writes invalidate rather
than update, Postgres first.** Redis evicts under memory pressure by design, so
it can never be the store. Invalidating (rather than rewriting) means the
post-append history is computed in exactly one place; deleting cannot disagree
with Postgres. Ordering is load-bearing — invalidating first leaves a window
where a concurrent read re-caches pre-append history for a whole TTL, and a test
pins the order. A Redis failure degrades to a Postgres read and is **the one
sanctioned exception to fail-loud** in the codebase. Full reasoning: **ADR 0008**.

### How test-profile LLM mocking is enforced (the mechanism)

**Under the `test` profile a real Anthropic client cannot be constructed.** Two
independent guards, both keyed on the *profile*, neither on the key's presence:

1. **`build_llm_client(settings)`** returns `ScriptedLLMClient` when
   `settings.is_test`. `create_app` goes through this, so `AnthropicClient.__init__`
   is never reached.
2. **`AnthropicClient.__init__` raises `RuntimeError` when `settings.is_test`** —
   *before the key is read*. This is the load-bearing one: it holds for any code
   that reaches past the factory, so no import order, fixture order, or
   monkeypatch can turn a unit test into a paid call.

Keying on the profile rather than on `anthropic_api_key is None` is the whole
point: a developer with `ANTHROPIC_API_KEY` exported (as I have — 108 chars, and
the suite still never called out) gets CI's exact hermetic suite.

Only the **network** is substituted. The graph, tool registry, orchestrator,
conversation store, route and SSE framing are the real thing in every test.

Pinned by `tests/unit/test_llm.py::TestTestProfileCannotCallAnthropic`, which
asserts the mechanism — including that a real key in the actual OS environment
changes nothing. Full reasoning: **ADR 0009**.

## 6. Deviations from the stage prompt — and why

### 1. The routes changed (the prompt said they should not)

The prompt says *"Routes and SSE plumbing do not change"* **and** *"populate
actual input_tokens/output_tokens ... replace, don't wrap"*. These conflict.
Stage 2 computed `usage` **in the route** by splitting the reply on whitespace.
The route cannot know what a turn cost — an agent run spans several model calls
it never sees — so honouring the token requirement forces usage to move to the
engine.

**What changed:** `CompletionEngine.complete` returns a `Completion` (text +
usage + finish_reason) instead of `str`; `stream` yields text pieces then one
terminal `Completion`; the route reads those.

**What did not change:** the endpoint, request/response schemas, SSE frame
format, `data: {json}` framing, the `[DONE]` sentinel, chunk shape, finish-reason
values. A Stage 2 client sees an identical wire format. Recorded in ADR 0006.

### 2. `langchain` was dropped, not promoted

The prompt says promote *"`langgraph`, `anthropic`, plus `langchain-anthropic` if
the chosen integration needs it"*. The existing extra also contained `langchain`,
which nothing imports. Pinning a dependency the code does not use makes the
lockfile fiction, so I removed it. `langchain-anthropic` was not needed. ADR 0006.

### 3. One Stage 2 test was rewritten, not deleted

`test_max_tokens_truncates_and_reports_length_finish_reason` asserted that
`max_tokens: 2` chops the reply to two whitespace words and reports
`finish_reason: "length"`. That behaviour was an artefact of `EchoEngine`'s
invented token counting — exactly what this stage replaces. `max_tokens` is now
an API parameter and `length` means the *model* reported `stop_reason:
"max_tokens"`.

I rewrote it as `test_finish_reason_is_length_when_the_model_hits_the_token_cap`,
which pins the real mapping, and added tests that `max_tokens` is forwarded and
that a default exists. **This is not editing a test to make it pass** — the
contract it described was deliberately removed by the stage — but it is a
judgement call and it is flagged here rather than buried.

Four `test_architecture.py` tests were removed for the same reason: they pinned
the Mermaid-CDN import and its "you need internet access" notice, which ADR 0010
deletes. They were replaced by tests asserting the property that made them
unnecessary (no external URLs, no `<script>`, inline SVG).

### 4. The diagram instruction contradicted the repo; I asked

The task said *"Diagrams stay pre-rendered static SVG, no CDN dependency
(existing convention — keep following it)"*. **No such convention existed** —
Stage 2 loaded Mermaid from a pinned CDN, and commit `8bc012a` had just added a
fail-loud notice for when that CDN is unreachable. I raised this rather than
guess; you chose pre-rendered inline SVG. Implemented per ADR 0010, which
supersedes the Stage 2 approach.

### 5. Two extra ADRs

The prompt required four (LangGraph design, migrations, caching, test mocking).
I wrote five — ADR 0010 covers the diagram change, which is a non-trivial
decision that supersedes prior work and warranted its own record.

### 6. An unplanned fix: a real circular import (found by the live call, not the tests)

`services/orchestrator/base.py` imported `services.api.schemas.ChatMessage`.
Because `services/api/__init__.py` built the whole app on import, a plain
`import services.orchestrator` **failed outright** — but only when nothing had
imported `services.api` first, which `conftest.py` always does. **The entire
suite passed while a plain script crashed.** I only hit it running the live check.

Fixed properly rather than papered over:

- `services/orchestrator/conversations.py` now owns a `Turn` domain type; the
  orchestrator no longer imports the API at all (the dependency runs
  api → orchestrator).
- `OrchestratorEngine` converts `ChatMessage` → `Turn` at the API boundary.
- `services/api/__init__.py` no longer builds the app on import.
- Four regression tests in `test_architecture.py::TestLayering` pin the rule via
  AST inspection, plus a subprocess check that `import services.orchestrator`
  works cold.

## 8. Known limitations, risks and incomplete pieces

**Do not read past this section thinking the stage is flawless.**

### Risks I would want a reviewer to look at first

1. **The hermetic suite cannot catch a wrong belief about the Anthropic API.**
   `ScriptedLLMClient` encodes my assumptions; if one is wrong, the fake and the
   code are wrong together and the tests still pass. Mitigated *once*, today, by
   the two real calls in §4 — but there is **no contract test**, so if Anthropic
   changes a field name, nothing in CI notices. (ADR 0009.)
2. **`temperature` is silently inert.** Accepted by the schema, never forwarded,
   because Opus 4.7+ rejects sampling parameters with a 400. A client setting it
   gets no effect and no error. Documented in the field and ADR 0006, but it is a
   wart.
3. **`/ready` does not check schema version.** `_migrate` deliberately never
   raises (ADR 0005's crash-loop reasoning), so a failed migration leaves the
   service reporting **ready** while every chat query fails. The honest fix is a
   readiness check on schema version; not built.
4. **No per-conversation concurrency control.** Two concurrent turns on one
   `conversation_id` both read the same history and both append; the unique index
   prevents corrupt ordering, but the second write **fails loudly** rather than
   serialising. (ADR 0008.)
5. **No context compaction.** A long enough conversation will exceed the context
   window and start failing. No prompt caching either — the system prompt and
   tool specs are re-sent every call.

### Not built, deliberately

- No retry/circuit breaking around the Anthropic call beyond the SDK's defaults
  (2 retries on 429/5xx) — Stage 9.
- The cache TTL (300s), pool sizes and `agent_max_steps` (6) are **untuned
  guesses**; no load data exists.
- The whole conversation history is cached as one Redis value — fine at current
  sizes, poor for a very long conversation.
- A step-capped run returns the model's last text, which may be a preamble to a
  tool call that never ran. Logged as a warning.
- `ScriptedLLMClient` ships in the image (never constructed outside `test`) — the
  price of enforcement-by-construction.

### Not verified this session

- **The Docker Compose stack was not booted end to end.** I ran `postgres` and
  `redis` (for the integration tests) but did **not** rebuild and run the `api`
  container, nor `scripts/smoke_health.sh`, nor Prometheus/Grafana. The Docker
  daemon was not running when I started and I brought up only what the tests
  needed.
- **CI was not run.** The GitHub Actions workflow is unchanged and I have no
  evidence it passes with the new dependency set. It needs no `ANTHROPIC_API_KEY`
  by design (ADR 0009), but that is an argument, not an observation.
- **`scripts/verify.ps1` / `verify.sh` were not run** — I ran their constituent
  commands individually (§4).
- **The `prod` profile was never booted.** The extended validator is unit-tested;
  a real prod boot is not.
- `FakePgPool` **does not execute SQL**. The unit tests prove the right statement
  is issued with the right arguments; the *live* integration run (§4) is what
  proves Postgres accepts it.
- Load, concurrency and failure-injection: nothing. Stage 9.

### Left running

`docker compose up -d postgres redis` containers are **still up** on this machine
(ports 5432 / 6379), holding a real `conversations` schema. `docker compose down`
to stop them.

## 10. Prerequisites for Stage 4 (RAG)

All met:

- **The `ToolRegistry`** — a retrieval tool is a new `Tool`, not a new loop.
- **The `Datastore` seam** — Qdrant's `httpx` probe swaps for `qdrant-client`
  behind the same contract (ADR 0005).
- **Conversation persistence** and the migration runner — add `0002_*.sql`.
- **`build_llm_client`** — embeddings can follow the same hermetic pattern.

⚠️ **One thing Stage 4 must not inherit blindly.** Stage 3's tools are pure
functions of their arguments, which is *why* it is safe to feed tool results
straight back into the model. A retrieval tool returns **document text** —
potentially attacker-controlled — so prompt injection through tool results
becomes a live concern the moment Stage 4 lands. `services/agents/README.md` and
`tools.py` both say so at the point where someone will add it.
